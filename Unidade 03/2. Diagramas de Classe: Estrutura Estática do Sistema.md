⚡**Os diagramas de classe nos fornecem a estrutura detalhada de como os dados e comportamentos estão organizados.** 

🎈Pressman e Maxim (2021) informam:

*Um diagrama de classe é uma representação visual do modelo de objetos do sistema (as classes que existem, seus atributos, seus métodos e como se relacionam).*

**Uma classe é um modelo (ou template) que descreve objetos de um tipo semelhante.**

---

🔨 **Atributos - Descrevendo o Estado:**

**Os atributos são listados na segunda seção da caixa que representa a classe.**

Um atributo tem três informações essenciais: 

- Visibilidade: Indica quem pode acessar esse atributo (+ para público, - para privado, # para protegido). Um banco de dados representado em um diagrama pode ter um atributo privado (-) “senha_admin” porque apenas a classe interna deve acessá-la, nunca o exterior.

- Nome: Nome do atributo.

- Tipo: Tipo de Atributo (int, bool, string).

---

🔨 **Métodos - Descrevendo o Comportamento:**

⚡**Os métodos descrevem as ações que os objetos da classe podem realizar.** 

Um método tem assinatura completa: visibilidade, nome, parâmetros e tipo de retorno.

🎈Page-Jones (2010) ressalta:

⚡**Os métodos revelam as responsabilidades da classe.** 

Se uma classe tem vinte métodos, provavelmente está fazendo muito e deveria ser decomposta. Se tem apenas um, talvez esteja subdimensionada. 

O número e a natureza dos métodos comunicam a coesão da classe – quanto mais relacionados estão os métodos em seu propósito, mais coesa é a classe.

---

🔨 **Relacionamentos - Conectando Estruturas**

⚡Três tipos principais de relacionamentos aparecem em diagramas de classe. 

- Associação: É o relacionamento mais geral, indicando que duas classes trabalham juntas.

- Herança: Permite reutilizar código e estabelecer hierarquias lógicas

No diagrama: Aparece como uma seta com ponta aberta apontando da classe filha para a classe pai.

- Composição: É uma relação forte - Uma parte não vive sem o todo.

Exemplo: Se você destruir um Livro, os Capítulos somem junto.

No diagrama: Usa um losango preenchido (♦).

- Agregação: É uma relação fraca - As partes são independentes.

Exemplo: Um Autor tem Livros, mas se o autor sumir, os livros continuam existindo.

No diagrama: Usa um losango vazio (♢).

🔨 **Padrões de Design e Estrutura de Classes:**

Compreender como organizar classes não é apenas uma questão técnica, é uma questão de design. Padrões de design clássicos (como Strategy, Observer, Factory) sugerem formas de estruturar relacionamentos entre classes para resolver problemas recorrentes.

---

🔨 **Diagramas de Sequência - Capturando Interações Dinâmicas:**

**Enquanto diagramas de classe mostram a estrutura estática, diagramas de sequência mostram como os objetos interagem ao longo do tempo para realizar uma tarefa.**

Sommerville (2003) enfatiza:

Diagramas de sequência são particularmente valiosos para documentar fluxos complexos, identificar problemas de lógica e comunicar o comportamento do sistema de forma que até não-programadores possam entender.

Um diagrama de sequência tem dois eixos: 

- Vertical (tempo fluindo de cima para baixo)

- Horizontal (diferentes atores ou objetos participantes)

- As linhas verticais com nomes representam os participantes. 

- As setas entre elas representam mensagens (chamadas de método, requisições) que os participantes trocam.

**O diagrama de sequência fica muito mais poderoso quando adicionamos blocos que controlam o caminho que o sistema vai seguir:**

- Alternativas (alt / if-else): Divide o desenho para mostrar caminhos diferentes.

Exemplo: 

- Se a senha estiver certa, o sistema entra.

- Se estiver errada, o sistema bloqueia e mostra um erro.

- Loops (loop): Mostra que uma ação vai se repetir várias vezes enquanto uma condição for verdadeira (como ler todos os itens de um carrinho de compras).

🎈COHN (2011) enfatiza:

*Isso permite modelar lógica condicional sem precisar desenhar múltiplos diagramas.*

🔨 **Diagramas de Atividades - Modelando Processos e Decisões:**

**Se diagramas de sequência focam em como objetos interagem, diagramas de atividades focam em fluxos de controle – o que acontece, em que ordem, e onde ocorrem decisões.**

🎈Campos (2014) informa:

*Diagrama de atividades são particularmente úteis para processos de negócio complexos, algoritmos com muitas ramificações e workflows.*

Um diagrama de atividades usa símbolos específicos: 

- Círculos pretos para início e fim.

- Retângulos arredondados para atividades.

- Losangos para decisões.

- Barras horizontais para sincronização.

- As setas indicam o fluxo do processo.

🔨 **Swimlanes (Raias) em Diagramas de Atividades:**

Quando múltiplos atores ou sistemas participam de um processo, use Swimlanes para separar responsabilidades: 

- Uma raia para "Cliente"
  
- Outra raia para "Sistema de Pagamento"
  
- Outra raia para "Armazém"

⚡**Isso torna imediatamente claro quem é responsável por cada atividade e onde ocorrem trocas entre sistemas.**

Essa visualização é crítica para comunicação com não técnicos, pois mostra explicitamente quem faz o quê.

---

🔨 **Integrando Modelagem Estrutural e Comportamental:**

- A modelagem C4 fornece a arquitetura. 

- Os diagramas de classe detalham a estrutura interna. 

- Os diagramas de sequência mostram como as classes colaboram. 

- Os diagramas de atividades documentam os processos que atravessam toda essa estrutura.

⚡**Considere um ciclo realista:**

Durante o design arquitetural (C4), você percebe que precisa de um novo componente. 

Você desenha diagramas de classe para detalhar sua estrutura. 

Para validar que os componentes se comunicam corretamente, você desenha diagramas de sequência. 

Quando percebe que o processo é complexo, com muitas decisões, você documenta com diagrama de atividades.

🎈Pressman e Maxim (2021) enfatizam:

*Cada modelo contribui com uma perspectiva diferente.*
