🔨 **Modelo C4:**

Criado por Simon Brown, propõe uma hierarquia de diagramas que varia de uma visão macro (contexto geral) até o nível de código implementado.

**O C4 é a definição de que toda arquitetura de software pode ser representada em quatro níveis progressivamente detalhados.**

⚡**A abordagem do C4 une diálogo com lógica do problema de comunicação arquitetural, o que faz ser tão poderosa.**

---

🔨 **O que Diferencia C4 de Outros Modelos?**

O modelo C4 não substitui UML ou outras notações; ele as complementa oferecendo uma estrutura hierárquica que ajuda a lidar com a complexidade arquitetural. Enquanto UML é mais detalhada e técnica, C4 é mais pragmática e comunicativa. Compreender quando usar cada uma é uma habilidade crítica na modelagem de sistemas modernos. 
Para aprofundamento, consulte a documentação oficial. 

---

🔨**Nível 1 - O Diagrama de Contexto do Sistema:**

Responde a uma pergunta essencial: 

**Qual é o meu sistema, e com o que ele interage?** 

Mostra o sistema como uma caixa única e fechada, focando apenas no contexto geral. 

**O objetivo é responder:**

O que é o sistema e quem (usuários ou sistemas externos) interage com ele, sem detalhar como ele funciona por dentro.

**O diagrama de contexto define o escopo do projeto.**

Ao mostrar claramente o que está dentro e fora do sistema, você alinha as expectativas com o cliente e a liderança, evitando cobranças futuras por funcionalidades que nunca fizeram parte do combinado.

---

🔨 **Nível 2 - O Diagrama de Contêiner:**

O diagrama de contêiner se refere a unidades de implantação separadas que compõe o sistema.

Exemplo:

- Um aplicativo web.

- Um banco de dados.

- Um aplicativo mobile.

- Um servidor de autenticação.

**Cada um desses é um contêiner**

🔨 **Nível 3 - O diagrama de Componente:**

É o nível que "abre" um contêiner (como uma API) para mostrar os blocos lógicos que o compõem. 

Exemplo: 

- Módulos de Autenticação.

- Carrinho.

- Pagamento.

**O que mostra:** 

- Como essas funcionalidades internas se relacionam, suas dependências e suas interfaces.

**Para que serve:** 

- Ajuda desenvolvedores e arquitetos a organizarem o código, controlarem o acoplamento (dependências) e garantirem que o sistema seja fácil de manter, reutilizar ou substituir.

---

🔨 **Nível 4: Código**

É o código-fonte real: 

- Classes.
- Métodos.
- Atributos.
- Estruturas de dados.

⚡**Tudo o que se torna necessário para o sistema funcionar.**

**O problema comum:**

Muitas equipes pulam os níveis anteriores e vão direto para o código. O resultado é um sistema que funciona localmente, mas cuja arquitetura geral ninguém entende.

---

🔨 **A solução do C4:**  

Execute em ordem de cima para baixo:

1. Primeiro você define o contexto, os contêineres e os componentes.

2. Escreve o código respeitando esses limites.

3. No final, o código final reflete perfeitamente os diagramas planejados.
