🔨 **Requisitos Funcionais - O que o Sistema Deve Fazer:**

**Os requisitos funcionais descrevem as funcionalidades específicas que o sistema deve fornecer.**

🎈Sommerville (2018) informa:

*São declarações sobre serviços que o sistema deve prover, como ele deve reagir a entradas específicas e como deve se comportar em determinadas situações.*

Exemplo - **Biblioteca universitária:**

1. O sistema deve permitir que os usuários realizem buscas por livros utilizando título, autor, ISBN (Código número que funciona como um CPF do livro) ou
palavras chave.

2. O sistema deve registrar empréstimos associando o livro ao usuário e definindo data de devolução.

3. O sistema deve enviar notificações automáticas por e-mail três dias antes do vencimento do prazo de devolução.

4. O sistema deve calcular automaticamente multas por atraso na devolução.

5. O sistema deve permitir renovações de empréstimo, desde que não haja reservas para o mesmo livro.

6. O sistema deve gerar relatórios de livros mais emprestados por período.

Cada requisito descreve uma ação ou comportamento específico do sistema.

**A clareza na redação de requisitos funcionais é fundamental. A interpretação diferente do especifiicado podem levar a erros durante o desenvolvimento,
consequentemente, um produto que não atende às expectativas.**

---

🔨 **Requisitos Não Funcionais - Como o sistema deve ser:**

Os requisitos não funcionais definem restrições e qualidades que o sistema deve possuir, incluindo:

1. Desempenho.

2. Segurança.

3. Confiabilidade.

4. Usabilidade.

5. Manutenibilidade.

🎈Sommerville (2018) observa:

*Requisitos não funcionais são frequentemente mais críticos que requisitos funcionais individuais, pois a falha em atender um requisito
não funcional pode tornar o sistema inútil.*

Exemplo - **Biblioteca universitária:**

Desempenho: O sistema deve processar buscas e retornar resultados em até dois segundos para 95% das consultas.

Disponibilidade: O sistema deve estar disponível 99,5% do tempo durante o horário de funcionamento da biblioteca.

Segurança: O sistema deve criptografas todas as senhas dos usuários utilizando o algoritmo bcrypt com salt.

Usabilidade: Usuários com formação básica devem ser capazes de realizar empréstimos sem treinamento prévio.

Escalabilidade: O sistema deve suportar até 10.000 usuários simultâneos durante períodos de pico.

Compatibilidade: O sistema deve funcionar nos navegadores Chrome, Firefox, Safari e Edge nas duas últimas versões.

**Esses requisitos não descrevem funcionalidades específicas, mas características qualitativas que o sistema precisa aprensentar.**

---

🔨 **Mensurabilidade de Requisitos não funcionais:**

Um erro comum na especificação de requisitos não funcionais é utilizar termos vagos. Esses termos são subjetivos e não mensuráveis.

**Todo requisito não funcional deve incluir métricas objetivas que permitam sua verificação.**

---

🔨 **Interação entre Requisitos funcionais e não funcionais:**

Decisões tomadas para atender requisitos não funcionais frequentemente impactam a implementação de requisitos funcionais.

Exemplo:

Atender a requisitos rigorosos de segurança pode adicionar complexidade à implementação de funcionaliodades de autenticação e autorização.

Requisitos de desempenho podem influenciar decisões arquiteturais, como o uso de cache ou otimização de consultas ao banco de dados.

🎈Audy(2015) e Pinho (2023) destacam:

*Em ambientes ágeis, a priorização de requisitos não funcionais deve acontecer continuamente.*

*Aspectos como segurança, desempenho e escalabilidade devem ser considerados desde as primeiras iterações, evitando 
retrabalho significativo posteriormente.
